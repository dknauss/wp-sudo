<objective>
Implement the Session Activity Dashboard Widget feature (v2.15.0) across three layers:
Event_Store (persistence), Event_Recorder (hook→store bridge), and Dashboard_Widget (UI).

Purpose: Give site administrators fast operational visibility into sudo session
activity directly from the WordPress dashboard. This is the first feature that
persists audit data — all 10 hooks currently fire-and-forget.

The feature ships as a single plan because the three layers are tightly coupled
and must be wired together before any layer delivers user-visible value.

This plan intentionally follows the leaner MVP defined in `docs/ROADMAP.md`,
not the broader first draft. The widget should establish the smallest durable
audit persistence layer needed for operator visibility without turning WP Sudo
into a general-purpose activity log product.
</objective>

<architecture>

## Three-Layer Design

### Layer 1: Event_Store (`includes/class-event-store.php`)

Thin wrapper around a custom table. All methods are static for simplicity
(same pattern as Sudo_Session's meta operations).

**Table:** `{$wpdb->base_prefix}wpsudo_events`

| Column     | Type              | Notes                                |
|------------|-------------------|--------------------------------------|
| id         | bigint unsigned   | AUTO_INCREMENT PRIMARY KEY           |
| site_id    | bigint unsigned   | Current site/blog ID for the event   |
| user_id    | bigint unsigned   | 0 for anonymous/system events        |
| event      | varchar(50)       | Hook name without `wp_sudo_` prefix  |
| rule_id    | varchar(100)      | Action Registry rule ID or empty     |
| surface    | varchar(30)       | admin, rest, cli, cron, xmlrpc, etc. |
| ip         | varchar(45)       | IPv4 or IPv6                         |
| context    | longtext          | JSON blob for optional metadata      |
| created_at | datetime          | UTC timestamp, DEFAULT CURRENT_TIMESTAMP |

**Indexes:** PRIMARY (id), INDEX (event, created_at), INDEX (site_id, created_at), INDEX (user_id, created_at)

**Static methods:**
- `create_table(): void` — runs `dbDelta()` with the schema
- `insert( array $data ): bool` — validates + inserts a row
- `recent( int $limit = 10, ?string $event = null ): array` — most recent N rows
- `count_since( string $event, int $seconds ): int` — count events in time window
- `prune( int $days = 14 ): int` — delete rows older than N days, return count
- `drop_table(): void` — for uninstall cleanup

### Layer 2: Event_Recorder (`includes/class-event-recorder.php`)

Subscribes only to the small event subset the widget needs. This follows the
canonical roadmap guidance to persist only the events that produce immediate
operator value in the dashboard.

**Hooks subscribed (5):**

| Hook                      | Args | Stored event name   |
|---------------------------|------|---------------------|
| wp_sudo_lockout           | 3    | lockout             |
| wp_sudo_action_gated      | 3    | action_gated        |
| wp_sudo_action_blocked    | 3    | action_blocked      |
| wp_sudo_action_allowed    | 3    | action_allowed      |
| wp_sudo_action_replayed   | 2    | action_replayed     |

Each callback maps hook arguments to an `Event_Store::insert()` call.
Payload structure follows the existing bridge plugins for consistency.

### Layer 3: Dashboard_Widget (`includes/class-dashboard-widget.php`)

Registered via `wp_dashboard_setup` hook. Visible to users with
`manage_options` capability.

**Three render sections:**

1. **Active Sessions** — queries user meta `_wp_sudo_expires` for all users
   with a future expiry. Shows count + top 5 usernames + time remaining.
   Data source: `get_users()` with meta_query, not the events table.

2. **Recent Events** — last 10 events from `Event_Store::recent()`.
   Columns: time ago, user, event label, rule ID, surface. On multisite, the
   table schema records `site_id`, but the first widget UI remains local/current-site
   focused rather than attempting a full network report.

3. **Policy Summary** — reads `wp_sudo_settings` option. Shows session
   duration and entry-point policy for each surface (REST, CLI, Cron,
   XML-RPC, WPGraphQL).

## Bootstrap and Lifecycle

- **Plugin::init()** creates `Event_Recorder` and `Dashboard_Widget` instances
- **Upgrader** gains `'2.15.0' => 'upgrade_2_15_0'` entry calling
  `Event_Store::create_table()`
- **Plugin activation** also calls `Event_Store::create_table()` for fresh installs
- **Cron:** `wp_sudo_prune_events` daily event calls `Event_Store::prune(14)`
- **Uninstall:** `uninstall.php` gains `Event_Store::drop_table()` call
  (or inline `$wpdb->query("DROP TABLE IF EXISTS ...")`)
