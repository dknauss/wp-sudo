<task type="auto">
  <name>Task 2: Create .wp-env.json and verify WordPress environment starts</name>
  <files>.wp-env.json</files>
  <action>
Create `.wp-env.json` at repo root:
```json
{
    "core": null,
    "plugins": ["."],
    "phpVersion": "8.2",
    "port": 8889,
    "config": {
        "WP_DEBUG": true,
        "WP_DEBUG_LOG": true,
        "SCRIPT_DEBUG": true
    }
}
```

Key decisions (per roadmap):
- `"core": null` uses latest stable WordPress. Will be pinned to a specific WP version URL for visual regression baselines in Phase 7.
- `"plugins": ["."]` mounts the current repo as the plugin under test. wp-env auto-activates it.
- `"phpVersion": "8.2"` — pinned, not default. Matches CI integration test PHP version.
- `"port": 8889` — avoids conflict with local dev sites (Local/Studio use 80/443, wp-env default is 8888 which may conflict with other dev tools).
- `WP_DEBUG` and `SCRIPT_DEBUG` enabled for verbose error reporting during test development.

After creating the file, verify wp-env starts by running:
1. `npx wp-env start` (requires Docker running)
2. `curl -s -o /dev/null -w "%{http_code}" http://localhost:8889/wp-login.php` — should return 200
3. `npx wp-env run cli wp plugin list --format=csv` — should show `wp-sudo` as active

If Docker is not running or wp-env fails to start, that is acceptable for the plan — the file is still correct. Document the Docker requirement in the task output. The CI workflow (Plan 03) will handle Docker availability automatically.

After verification, run `npx wp-env stop` to clean up.
  </action>
  <verify>
Run: `cat .wp-env.json | python3 -c "import sys,json; d=json.load(sys.stdin); assert d['port']==8889; assert d['plugins']==['.']" && echo "wp-env config valid"`

Config file validates with correct port and plugin mount.

If Docker is available: `npx wp-env start && curl -s -o /dev/null -w "%{http_code}" http://localhost:8889/wp-login.php && npx wp-env stop` returns 200.
  </verify>
