    Files:  wp-sudo.php, tests/e2e/specs/multisite-network-admin.spec.ts,

            (for example `plugin_dir_url( __FILE__ )`) is safe even when

    Reality: PHP executes the plugin from the symlink target path, so
            naive bootstrap URL derivation can produce asset URLs like

            plugin basename from active plugin state and derive the URL
            from that recovered path.
    Source:  Local runtime trace from `multisite-subdomains.local`,
            PHP `__FILE__` symlink behavior repro, WordPress core
            `wp-includes/plugin.php` (`plugin_basename()`,
            `wp_register_plugin_realpath()`) and
            `wp-admin/includes/plugin.php` activation-path handling.
    Notes:  This was not a stale count or a typo. It was an unverified
            environment assumption that looked plausible in code review
            but failed immediately in a real symlinked multisite browser
            flow. The browser trace, not theory, exposed the defect.

ROOT CAUSE
----------
All errors have stemmed from patterns that boil down to *not checking the 
actual source*, which leads to stale or confabulated source material.

- Generating reference content from training data instead of verifying
against live, authoritative sources. This is now a requirement.

- Losing track of current state (workflow progress). STATE.md now tracks 
workflow progress: which phase, which plan, what's done.

- Losing track of current metrics (test and line counts). CURRENT-METRICS.md 
tracks this now. 

- Losing track of architectural facts — the volatile numbers that change
when features ship: total request surfaces, gated rules, settings fields,
dropdowns, help tabs. The "Architectural Facts" table in CURRENT-METRICS.md
tracks this now.

- Assuming environment semantics from memory instead of verifying them in
live runtime. Local/Studio symlinked plugin mounts can change how
bootstrap path helpers behave. When the task touches filesystem-backed
runtime behavior, verify against the actual environment, not against what
WordPress "usually" does.

The fabricated method and meta key names are the worst category — these
are not stale facts but invented strings that sound plausible. A human
reading the documentation would have no reason to doubt them, and code
using them would fail silently or throw errors.
